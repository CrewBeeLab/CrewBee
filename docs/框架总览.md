# AgentScroll 工程框架总览

## 1. 文档目的

这份文档用于把项目书中的抽象设计，翻译成当前仓库已经落地和即将落地的工程框架。它面向人类阅读，重点回答三个问题：

1. AgentScroll 当前的工程骨架是什么。
2. 各个目录和文件在整体方法论中分别承担什么角色。
3. 下一步开发应该从哪里开始，而不是重新设计一遍框架。

## 2. 当前框架总公式

当前工程遵循项目书中的 V1 收束公式：

`AgentScroll = Agent Team + Manager + Adapter`

也就是说，项目不是在做一个“自动决定一切”的多 Agent 平台，而是在做一套可移植、可切换、可管理的 Agent Team 框架。

## 3. 当前目录如何对应这个公式

### 3.1 `src/`

`src/` 是面向程序运行和后续开发的代码框架。

- `src/core`：定义 Team、Agent、Policy、Capabilities、Runtime 等核心类型。
- `src/agent-teams`：Team Library 子系统，负责把内置 `CodingTeam` 与文件型 Team 装配成运行时可直接消费的 Team Library。
- `src/manager`：负责 Team 选择、模式切换、初始执行计划生成、运行快照维护。
- `src/adapters`：负责把 Team 映射到宿主运行时的上下文与事件模型。

### 3.2 `AgentTeams/`

`AgentTeams/` 是“已经定义好的 Team 目录”。

这里存放的是当前项目通过配置文件定义的 Team 静态资产，例如：

- `GeneralTeam`
- `WukongTeam`

每个配置文件型 Team 目录至少包含：

- `team.manifest.yaml`
- `team.policy.yaml`
- `shared-capabilities.yaml`
- `agents/*.agent.md`
- `docs/TEAM.md`（或 `TEAM.md`）

这意味着：

- 代码层的 `src/agent-teams` 是运行时组合结果；
- 其中 `CodingTeam` 为内置 Team，不直接作为公开配置目录暴露；
- 文件层的 `AgentTeams/` 是其他公开 Team 的配置资产来源。

当前 `src/agent-teams` 内部已经进一步拆为：

- `parsers.ts`：纯解析和字段映射
- `filesystem.ts`：文件发现与目录加载
- `embedded/coding-team.ts`：内置 Team 源
- `documentation.ts`：文档引用整理
- `validation.ts`：Team / Library 校验
- `library.ts`：多 Team source 装配入口

这样做的目的，是让 Team source、解析规则、校验逻辑和运行时装配不再混在一个文件中，降低后续继续演化 OpenCode plugin / host adapter 时的改动爆炸半径。

### 3.3 `AgentTeamTemplate/`

`AgentTeamTemplate/` 是模板目录，用于新增 Team 时复用。

它不是一个真实业务 Team，而是一个“新 Team 应该长成什么样”的示例骨架。当前模板中保留了：

- Team manifest
- Team policy
- shared capabilities
- 示例 agents
- Team 说明文档

这样后续新增 Team 时，不需要再从零发明目录结构和字段组织方式。

### 3.4 `docs/`

`docs/` 是面向人类阅读的中文框架文档目录。

它的职责不是替代内部规范原文，而是把：

- 项目书中的方法论
- 当前代码框架中的真实落地结构
- 后续开发的进入点

三者串起来，降低理解和接手成本。

## 4. 当前 Team 文件模型

一个 Team 在当前框架中，被拆成以下几层静态资产：

### 4.1 `team.manifest.yaml`

负责回答：

- 这个 Team 是谁
- Leader 是谁
- 成员有哪些
- 默认 Workflow 是什么
- 运行模式有哪些
- 当前协作假设是什么

### 4.2 `team.policy.yaml`

负责回答：

- 团队共享规则是什么
- 哪些动作需要审批
- 质量底线是什么
- 哪些行为是明确禁止的

### 4.3 `shared-capabilities.yaml`

负责回答：

- 默认模型档位是什么
- 默认工具配置是什么
- 团队共享哪些 skills / instruction packs / hooks / memory / MCP

### 4.4 `agents/*.agent.md`

这是单个 Agent 的静态画像文件。它强调两个核心：

- `persona core`：这个 Agent 是怎样的做事者
- `responsibility core`：这个 Agent 负责什么、不负责什么

这使得 Agent 不再只是“一坨 prompt”，而是具有明确边界的结构化资产。

## 5. 当前运行假设

当前 V1 工程框架坚持以下假设：

1. 人类主要与 Leader 交互，而不是直接调度内部成员。
2. Team 是主入口，成员是 Team 内部结构。
3. Session Context 是当前阶段最主要的协作媒介。
4. 先把 Team 定义、切换、执行入口和管理骨架打稳，再考虑更复杂的自动路由与长期状态系统。

## 6. 当前已经完成的框架价值

到目前为止，项目已经不是“只有文档概念”的阶段，而是具备了下面这些可继续开发的基础：

- 有了清晰的 Team-first 类型框架
- 有了两个通过配置文件定义的 Team 目录，以及一个内置 `CodingTeam`
- 有了一个可复用的 Team 模板目录
- 有了 Manager 与 Adapter 的代码骨架
- 有了可直接继续实现 loader / validator / OpenCode adapter 的入口

## 7. 下一步最合理的开发顺序

基于当前框架，建议按下面顺序继续：

1. 在 Team Library 子系统之上补 runtime assembler，把 Team package + Mode 收敛为宿主可消费的 runtime blueprint。
2. 开始实现 OpenCode Adapter，把 Team runtime blueprint 映射到宿主插件接口。
3. 补 Manager 的 Team catalog、启用、校验与基础运行展示能力。
4. 为 Team source 的合并规则、安装来源和错误可视化补充更完整的管理面能力。

## 8. 一句话总结

当前 AgentScroll 已经完成了从“项目书概念”到“可执行工程框架”的第一段转换：

它现在是一套以 `Agent Team` 为一级对象、以混合 Team 源（内置 `CodingTeam` + 文件型公开 Team）为运行基础、以 `AgentTeamTemplate/` 为模板入口、以 `src/` 为运行时骨架的 V1 框架工程。
